## Code Review: /Users/mason/Documents/CODE/axhxrx-megarepo/utf8-stream-occurrences @ HEAD

1 finding (1 P2)

The implementation has a real UTF-8 decoding bug: initial BOM bytes are dropped even though the API and comments say they should be preserved. This causes incorrect counts for BOM-bearing streams and breaks equivalence between byte and string ingestion.

[P2] Preserve BOM bytes at the start of a new decoder stream
     /Users/mason/Documents/CODE/axhxrx-megarepo/utf8-stream-occurrences/src/utf8ChunkDecoder.ts:23-25
     `ignoreBOM: false` has the opposite effect here: the first UTF-8 BOM in each fresh `TextDecoder` stream is stripped instead of being emitted. That means `appendBytes(new Uint8Array([0xEF,0xBB,0xBF,...]))` is not equivalent to `appendString('\uFEFF...')`, so patterns that include the BOM (for example `"\uFEFFfoo"`) can never match, and the same false negative recurs after every `end()` because the decoder is recreated.

Author notes (axhxrx-engineer agent, 2026-04-17):

Reviewer is correct. Confirmed via the WHATWG Encoding spec and MDN: `ignoreBOM: false` is "strip the BOM" (the opposite of what the flag name suggests). SPEC.md § UTF-8 decoding requires: "A BOM at the start of the first chunk is NOT stripped."

Fix: changed `ignoreBOM` to `true` in `src/utf8ChunkDecoder.ts` and added an inline comment explaining the counter-intuitive naming. Added two tests in `test/utf8ChunkDecoder.test.ts` (leading BOM preserved; BOM preserved again after `end()`) and one cross-check test in `test/OccurrenceCounter.test.ts` asserting that `appendBytes` of a BOM-prefixed UTF-8 sequence matches the equivalent `appendString` input. All tests pass under `bun test` and `deno test -A`.
